On the Misalignment between Theory and Practice: A Study of Refactoring Customizations


This website contains complementary data to the paper.

Abstract

Background:Refactoring is a common software development practice with books defining standard mechanics of popular refactoring types. There are also IDEs with different implementations for those common refactoring types. However, it is unclear whether (i) developers follow standard mechanics presented in refactoring books, and (ii) IDEs offer tooling support for refactorings performed in practice. To answer these questions, we conduct an analysis of 1,162 refactorings composed of more than 100k modifications from 13 real software projects. Our results reveal that developers often apply recurring additional modifications besides the standard ones in their refactorings, a.k.a. customized refactorings. Customized refactorings contain additional modifications not covered by the most popular refactoring book. For instance, we found this is the case of 80.77% of the Move Method instances observed in software projects. We also study the refactoring tools of popular IDEs and observed that several customizations often applied by developers are not fully supported. Additionally, to understand the relevance of these customizations, we conducted a survey with 41 developers about the most frequent customizations we found. Developers confirm they apply those customized refactorings in practice and agree proper tool support is required to perform them. Our results suggest this is one of the key reasons why developers do not widely use tools to apply refactorings. Moreover, refactoring guidelines must be updated to reflect typical refactoring customizations. Update of these guidelines can help IDE builders to improve refactoring support for developers applying their customizations. Our contributions include customization patterns applied by developers that serve as the basis for updating guidelines and improving tool support.

Applicability

Our study complements previous studies[8][14] and helps the development of refactoring tools well aligned with refactorings in practice. Tool developers can use these catalogs to create flexible tools, allowing developers to apply the patterns as a configuration of the tool or through recommendations. For instance, the following image illustrates an additional configuration for Netbeans. This new configuration should allow developers to handle the client methods that will be affected by the refactoring, especially, to handle the modification *Method Access* that composes the most frequent patterns. In this way, developers should be able to decide how each client method should be affected.
For more complex patterns, composed of several modifications, refactoring tools should allow developers to create their own customization in a step-wise way, adding or removing code modifications that compose each refactoring. Finally, these tools could also recommend customized refactorings for each developer.
